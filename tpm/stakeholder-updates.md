# 📊 Stakeholder Update Prompts

**Role:** Technical Project Manager  
**Category:** Stakeholder Communication  
**Use when:** Sending weekly status updates, presenting to leadership, escalating risks, or announcing a launch.

---

## Prompt 1 — Weekly Status Report

```
You are a Technical Project Manager writing a weekly status report for leadership and key stakeholders.

Project name: [e.g., "Platform Migration v2"]
Reporting period: [e.g., Week of April 7–11, 2025]
Overall health: [Green / Yellow / Red]
Health reason: [1 sentence explaining the status color]

Progress this week:
- [What was completed or moved forward]
- [Any milestones hit]

Upcoming next week:
- [What the team is focused on]

Risks / blockers:
- [Active issues and their status]

Decisions needed from stakeholders:
- [Any open questions or approvals needed]

Please write a concise, professional status report (under 300 words) that:
1. Opens with a one-line project health summary
2. Uses plain language — no jargon
3. Highlights risks clearly without causing panic
4. Ends with a clear ask or next step for the reader
```

---

## Prompt 2 — Executive Summary (for Leadership / C-Suite)

```
You are a Technical Project Manager preparing a brief for senior leadership who have limited time.

Project: [name and one-line description]
Timeline: [start date → target end date]
Current phase: [e.g., "development", "UAT", "rollout"]
Key accomplishments to date: [list]
Current risks: [list]
Budget status: [on track / over / under — include % if known]
What you need from leadership: [approval / decision / visibility / nothing]

Write a 1-page executive summary that:
1. Leads with outcomes, not activity
2. Quantifies impact where possible (e.g., "reduces manual processing time by ~30%")
3. Presents risks with mitigation already attached
4. Has a clear, single call to action at the end
5. Tone: confident, direct, not defensive
```

---

## Prompt 3 — Risk Escalation to Leadership

```
You are a Technical Project Manager escalating a project risk to leadership.

Project: [name]
Risk description: [What is the risk and why does it matter?]
When it was identified: [date or sprint]
Current impact if unresolved: [timeline delay / cost / quality / customer impact]
What has already been tried: [list mitigation attempts]
What you need from leadership: [decision / resource / unblocking action]
Urgency: [e.g., "decision needed by [date] or we slip the launch by X weeks"]

Write a clear escalation message (email or Slack-ready) that:
1. States the risk in the first sentence
2. Does not assign blame
3. Provides two resolution options with trade-offs
4. Makes it easy for leadership to act with a single reply
```

---

## Prompt 4 — Launch / Go-Live Announcement

```
You are a Technical Project Manager announcing a product launch or feature go-live.

What launched: [feature / product / system name]
Launch date: [date]
Who it affects: [internal teams / customers / both]
Key benefits: [2–3 bullet points]
What changes for users: [what do they need to know or do differently?]
Support plan: [who to contact, where to file issues, FAQs location]
Thanks / shoutouts: [teams or individuals to recognize]

Write a launch announcement for [choose: company-wide Slack / email / Confluence page] that:
1. Leads with the "what's in it for me" for the audience
2. Is celebratory but not fluffy
3. Gives clear next steps for anyone affected
4. Includes a link or contact for questions
```

---

## Tips for Using These Prompts

- For Prompt 1 and 2: Keep a running notes doc during the week so you're not writing from memory.
- For Prompt 3: Escalate early — waiting makes the situation and the message worse.
- For Prompt 4: Draft the launch announcement before go-live day so it's ready to send immediately.
- All outputs can be pasted directly into Slack, email, Confluence, or Notion.
